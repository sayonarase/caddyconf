This file lists only changes that I think should be made. It intentionally
ignores sonnet-recommendations.txt and is based on an independent GPT-5.4 review.


1. CRITICAL: Path traversal in /api/save-config can write outside configs/
--------------------------------------------------------------------------------
File:
- server.py (save_config, around lines 69-82)

Problem:
- The filename is derived from hostname using only:
  safe_hostname = hostname.replace('*', '_wildcard_').replace(':', '_')
- This does not remove path separators or traversal segments like ../
- A crafted hostname such as ../../tmp/evil can escape CONFIGS_DIR and write
  arbitrary files the process has permission to write.

Why it matters:
- This is a serious file-write vulnerability.
- In Docker or on a server, it can overwrite application or system files.

What should change:
- Normalize and validate the final path with os.path.realpath/os.path.abspath.
- Reject any filename that does not stay inside CONFIGS_DIR.
- Prefer a strict allowlist for filenames based on expected hostname characters.


2. CRITICAL: Path traversal in /api/config-history-content can read arbitrary files
--------------------------------------------------------------------------------
File:
- server.py (config_history_content, around lines 124-133)

Problem:
- The route uses <path:filename>, then joins filename directly to history_dir.
- No containment check is done.
- A request like ../../server.py can escape configs/history and read local files.

Why it matters:
- This is a serious file-read vulnerability.
- It can expose application code, secrets, or anything readable by the process.

What should change:
- Resolve the final path and verify it stays inside history_dir.
- Restrict allowed filenames to expected backup names only.
- Require the filename to end with .bak.


3. HIGH: SFTP upload/download can escape the configured remote directory
--------------------------------------------------------------------------------
File:
- server.py (ssh_download_file and ssh_upload_file, around lines 671-717)

Problem:
- filename is joined to session['remote_path'] without validation:
  os.path.join(session['remote_path'], filename).replace('\\', '/')
- A filename like ../etc/caddy/Caddyfile can escape the chosen directory.

Why it matters:
- Any authenticated SSH user can read or overwrite files outside the selected
  config directory on the remote Linux server.
- That breaks the safety expectation of “choose a folder first, then work there”.

What should change:
- Use posixpath.normpath on the remote path.
- Verify the normalized path stays inside session['remote_path'].
- Reject filenames with path separators or traversal attempts.


4. HIGH: SSH host key verification is disabled
--------------------------------------------------------------------------------
File:
- server.py (ssh_connect, line around 577)

Problem:
- The code uses paramiko.AutoAddPolicy(), which accepts any host key.
- That means SSH credentials and config data can be sent to a spoofed host if
  there is a man-in-the-middle on the network.

Why it matters:
- This is a real security weakness, especially because the app accepts both
  passwords and pasted private keys.

What should change:
- Support host fingerprint verification in the UI/API.
- Or require known_hosts checking with RejectPolicy.
- At minimum, do not silently trust unknown hosts.


5. HIGH: ZeroSSL config generation is wrong and likely breaks certificate issuance
--------------------------------------------------------------------------------
File:
- public/js/config-builder.js (around lines 106-123)

Problem:
- The generated ZeroSSL EAB config uses the same value for both:
  key_id
  mac_key
- ZeroSSL EAB requires two distinct values.

Why it matters:
- Users selecting ZeroSSL or ZeroSSL + fallback will generate broken TLS config.
- Certificate issuance can fail even though the UI suggests the setup is valid.

What should change:
- Add a separate UI field for the ZeroSSL MAC key.
- Generate key_id from one field and mac_key from the other.
- Update i18n/help text to explain the difference.


6. MEDIUM: Many POST endpoints return 500 on invalid or missing JSON
--------------------------------------------------------------------------------
File:
- server.py (multiple endpoints using request.get_json())

Problem:
- Several endpoints call request.get_json() and immediately use data.get(...)
- If the body is missing, malformed, or has the wrong content type, data can be
  None and the endpoint crashes with AttributeError.

Why it matters:
- Users and API callers get misleading 500 errors instead of clean 400 errors.
- This affects reliability and makes troubleshooting harder.

What should change:
- After each request.get_json() call, check that data is not None.
- Return a consistent 400 JSON error such as:
  { "error": "Invalid or missing JSON body" }


7. MEDIUM: git-push accepts an unsanitized files list
--------------------------------------------------------------------------------
File:
- server.py (git_push, around lines 452-481)

Problem:
- files = data.get('files', []) is passed straight into:
  subprocess.run(['git', 'add'] + files, ...)
- shell=False avoids shell injection, but the input can still contain path
  traversal or unexpected git pathspec/flag-like values.

Why it matters:
- It weakens the trust boundary around what may be staged and pushed.
- It can stage files outside the intended config set or behave unpredictably.

What should change:
- Validate each filename with a strict allowlist.
- Reject path separators, traversal, and values beginning with -.
- Ideally only allow .caddy filenames from CONFIGS_DIR.


8. MEDIUM: Passive health-check latency uses the wrong Caddy directive
--------------------------------------------------------------------------------
File:
- public/js/config-builder.js (around lines 653-660)

Problem:
- passiveUnhealthyLatency is emitted as:
  unhealthy_request_count <value>
- But a latency value (e.g. 1s) should map to unhealthy_latency, not
  unhealthy_request_count.

Why it matters:
- The generated Caddyfile can be invalid or semantically wrong.
- This can prevent Caddy from loading the config.

What should change:
- Replace unhealthy_request_count with unhealthy_latency for that field.
- Or rename the field if the intended setting is actually request count.


9. MEDIUM: Docker image is labeled production but still runs the Flask dev server
--------------------------------------------------------------------------------
File:
- Dockerfile
- server.py

Problem:
- Docker sets FLASK_ENV=production, but CMD still runs:
  python server.py
- That starts Flask’s built-in development server, just with debug disabled.

Why it matters:
- The image looks deployment-ready but is not using a production WSGI server.
- Under load or public exposure, this is less robust and less safe than using
  gunicorn/waitress/uvicorn-equivalent for Flask.

What should change:
- Add a production server such as gunicorn to requirements.txt.
- Run the container with gunicorn instead of python server.py.
- Update README and install docs accordingly if Docker is meant for real hosting.


10. MEDIUM: install.sh duplicates dependency definitions instead of using requirements.txt
--------------------------------------------------------------------------------
File:
- install.sh (around lines 187-192)

Problem:
- The installer hardcodes:
  pip install --quiet flask cryptography bcrypt paramiko
- requirements.txt already exists and is copied into the install directory.
- This has already drifted once in the project history and can drift again.

Why it matters:
- Users may install different dependencies than the app actually requires.
- Future updates can break installs if requirements.txt and install.sh diverge.

What should change:
- Use:
  pip install --quiet -r requirements.txt
- Keep requirements in one place only.
