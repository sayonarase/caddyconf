CRITICAL ISSUES
--------------------------------------------------------------------------------

1. PATH TRAVERSAL – /api/save-config writes files outside configs/
   File: server.py, around line 80-82
   Problem:
     safe_hostname = hostname.replace('*', '_wildcard_').replace(':', '_')
     Only * and : are stripped. Forward slashes and .. are left intact.
     A hostname like "../../tmp/evil" creates filepath outside configs/,
     allowing an attacker to overwrite arbitrary files in the container.
   Fix:
     After building filepath, assert it stays within CONFIGS_DIR:
       filepath = os.path.join(CONFIGS_DIR, filename)
       if not os.path.realpath(filepath).startswith(os.path.realpath(CONFIGS_DIR)):
           return jsonify({'error': 'Invalid hostname'}), 400

2. PATH TRAVERSAL – /api/config-history-content reads arbitrary files
   File: server.py, around line 124-133
   Problem:
     Route uses <path:filename> which allows / in the parameter.
     No validation exists. GET /api/config-history-content/../../server.py
     returns the server source code. With enough ../ any readable file
     on the container is exposed (including any pasted SSH keys).
   Fix:
     Add containment check:
       if not os.path.realpath(filepath).startswith(os.path.realpath(history_dir)):
           return jsonify({'error': 'Invalid filename'}), 404
     Also require filename to end with '.bak'.

3. PATH TRAVERSAL – SFTP upload/download allows escaping remote_path
   File: server.py, around line 687 and 712
   Problem:
     filename is joined to remote_path without validation.
     Submitting filename="../etc/caddy/Caddyfile" lets an authenticated
     user read or overwrite ANY file the SSH user has access to on the
     remote server, not just files in the configured directory.
   Fix:
     import posixpath
     remote_file = posixpath.normpath(posixpath.join(session['remote_path'], filename))
     if not remote_file.startswith(session['remote_path'].rstrip('/') + '/'):
         return jsonify({'error': 'Invalid filename'}), 400

4. SSH MITM VULNERABILITY – AutoAddPolicy accepts any host key
   File: server.py, around line 577
   Problem:
     ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
     silently accepts any host key, including one from a man-in-the-middle.
     Passwords and private keys are transmitted and could be intercepted.
   Fix:
     Accept an optional 'host_fingerprint' in the connect request body.
     If provided, verify it. Otherwise warn the user in the UI that the
     connection is not fingerprint-verified. As a minimum, replace
     AutoAddPolicy with WarningPolicy.

5. ZEROSSL BUG – key_id and mac_key are set to the same value
   File: public/js/config-builder.js, around lines 110-111 and 122-123
   Problem:
     Both zerossl_api and zerossl_fallback blocks write:
       key_id  <zerosslApiKey>
       mac_key <zerosslApiKey>
     ZeroSSL EAB requires TWO different values: a short key_id and a
     separate base64-encoded mac_key (HMAC secret). They are never
     the same string. Any config generated with ZeroSSL will fail
     silently – Caddy cannot obtain certificates.
   Fix:
     Add a second input field "ZeroSSL MAC Key" (id="zerosslMacKey")
     next to the existing API key field. Use zerosslApiKey for key_id
     and zerosslMacKey for mac_key.

6. NULL DEREFERENCE – All POST endpoints crash on invalid JSON
   File: server.py, multiple endpoints (lines 72, 166, 229, 438, 452,
         562, 630, 648, 674, 698, 723, 743 and others)
   Problem:
     Every endpoint calls data = request.get_json() then data.get(...)
     without checking if data is None. get_json() returns None if:
     - Content-Type is not application/json
     - Body is empty
     - Body is malformed JSON
     This raises AttributeError → unhandled 500 error.
   Fix:
     Add after every get_json() call:
       if not data:
           return jsonify({'error': 'Invalid or missing JSON body'}), 400
     Or use: request.get_json(force=True, silent=True)

7. COMMAND INJECTION RISK – user-controlled 'files' list passed to git add
   File: server.py, around line 478-479
   Problem:
     files = data.get('files', []) is passed unsanitized to subprocess
     as git add arguments. While shell=False prevents OS injection,
     Git interprets its own flags – an entry like '--exec=...' could
     trigger git hooks. Entries with ../ could stage files outside
     configs/ and leak them into git history.
   Fix:
     Validate each entry: only allow alphanumeric characters plus
     . - _ and no path separators. Reject the request if any entry
     fails validation.

8. WRONG CADDY DIRECTIVE – passive health check latency field
   File: public/js/config-builder.js, around lines 653 and 660
   Problem:
     The field passiveUnhealthyLatency (a duration like "1s") is
     emitted as the directive:
       unhealthy_request_count <value>
     But unhealthy_request_count expects an INTEGER (number of
     concurrent requests), not a duration. This produces a Caddyfile
     syntax error, preventing Caddy from loading the config entirely.
   Fix:
     Change the emitted directive to:
       unhealthy_latency <value>
     Or rename the field to accept an integer and keep
     unhealthy_request_count.
